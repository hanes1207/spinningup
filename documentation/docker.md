# Docker Container
**`docker run [옵션] [이미지 이름] [실행할 명령]`** (create는 생성만, run은 생성하고 실행) \
 -d: detached 모드(백그라운드 실행), 컨테이너를 뒷단에서 계속 실행할 때 필수 \
 -it: Interactive + TTY, 컨테이너 내부 터미널로 접속할 때 사용 \
 --name: 컨테이너 이름 설정 \
 -p: port forwarding, 8080:80은 외부 8080 포트를 내부 80포트로 연결 \
 -v: 데이터 저장, 호스트의 디렉토리를 컨테이너와 공유 (데이터 휘발 방지) \
 -e: 환경변수 설정 \
 --rm: 일회성 실행, 컨테이너 종료 시 자동으로 삭제 (테스트용으로 유용) \
 --network: 네트워크 연결, 특정 도커 네트워크에 컨테이너를 소속시킴

**`docker start [컨테이너 이름 or ID]`**: 단순히 컨테이너를 다시 실행 \
컨테이너가 백그라운드(-d 모드)로 실행된다. \
컨테이너 내부의 프로세스가 다시 시작될 뿐, 새로운 설정(포트 포워딩, 볼륨 등)을 바꿀 수는 없다.

**`docker attach [컨테이너 이름]`**: 컨테이너의 **주 프로세스**(메인 실행 명령)에 다시 연결 \
attach로 들어갔다가 exit를 누르면 컨테이너 자체가 종료될 수 있음 (메인 프로세스가 종료되기 때문)

**`docker exec -it [컨테이너 이름] /bin/bash`**: 실행 중인 컨테이너 내부에서 새로운 터미널 세션을 하나 더 연다. \
작업하다가 exit를 입력해도 컨테이너는 계속 실행 상태를 유지한다.

**`docker stop [컨테이너 이름 또는 ID]`**: 실행 중인 프로세스에 SIGTERM 신호를 보내 안전하게 종료.

**`docker rm [컨테이너 이름 또는 ID]`**: 중지된 컨테이너를 제거.

## -p, -v, --network 설정은 못바꾸나? --> 맞다
도커 컨테이너의 설정은 생성 시점에 정의된 '컨테이너 명세(Configuration)'에 고정된다.

도커 컨테이너는 일종의 '불변(Immutable)' 객체로 설계되었다. \
컨테이너 내부의 파일이 바뀌는 것과, 컨테이너를 구동하는 환경 설정(포트, 네트워크, 볼륨)은 완전히 다른 개념.

컨테이너의 설정 정보는 도커 엔진 내부의 config.v2.json 같은 파일에 기록되는데, 이 파일은 컨테이너가 실행되는 동안 안전을 위해 외부 수정으로부터 보호됨.

### 설정을 변경하고 싶다면? --> 삭제 후 재생성 해야 함
```
# 1. 기존 컨테이너 중지 및 삭제
docker stop [컨테이너 이름]
docker rm [컨테이너 이름]

# 2. 원하는 설정으로 다시 run
docker run -d -p 9090:80 -v /my/data:/data --network my-net --name [컨테이너 이름] [이미지 이름]
```

삭제하고 다시 만들기는 너무 귀찮다면 \
데이터는 무조건 외부로 (-v 옵션): 컨테이너가 사라져도 데이터는 볼륨(Volume)에 남아있음. 따라서 컨테이너를 삭제하고 다시 만들어도, 같은 볼륨을 연결하면 데이터는 그대로 유지됩니다.

Docker Compose 사용:
docker-compose.yml 파일 하나에 설정을 다 적어두면, 설정 변경 후 docker-compose up -d 명령어 한 번으로 컨테이너를 알아서 내리고 다시 올려준다.

네트워크는 나중에 추가 가능: --network 옵션은 컨테이너를 삭제하지 않고도 docker network connect [네트워크이름] [컨테이너이름] 명령어로 나중에 붙일 수는 있습니다. 하지만 포트나 볼륨은 여전히 재생성이 필요합니다.

## 포트 설정 진짜 변경 불가한가
이미 생성된 컨테이너의 포트 매핑(-p)은 도커 엔진의 핵심 설정(Metadata)이라 직접 수정이 불가능. \
개발 환경이나 특정 상황에서 사용하는 우회적인 해결책들은 존재.

1. '공유 네트워크(Host Mode)' 사용 (가장 쉬운 우회) \
컨테이너 생성 시 --network host 옵션을 주면, 컨테이너가 호스트의 네트워크를 그대로 공유. \
장점: 컨테이너 내부에서 애플리케이션의 포트 설정만 바꾸면 된다. 컨테이너를 다시 만들 필요가 없다. \
단점: 포트 충돌이 발생할 수 있고, 도커의 네트워크 격리 기능을 포기하는 것.

2. 리버스 프록시(Nginx 등) 앞단에 배치 \
컨테이너 앞단에 Nginx 같은 웹 서버를 두고, 프록시 설정만 바꾸는 방식. \
흐름: 사용자 -> [Nginx:80] -> [컨테이너:8080] \
장점: 포트를 바꿔야 할 때 Nginx 설정만 수정하면 됩니다. 실무에서 컨테이너를 건드리지 않는 가장 권장되는 방법입니다.

3. 커밋(Commit) 후 재생성 (번거롭지만 데이터 보존) \
컨테이너 내부 파일 설정을 변경해서 새 이미지를 만든 뒤, 그 이미지를 기반으로 새 포트로 컨테이너를 올리는 방법. \
`docker commit [컨테이너] [새이미지명]` \
`docker run -p [바꿀포트]:80 [새이미지명]`




# Docker GUI setting (MacBook)
1. Install XQuartz in MacBook: `brew install --cask xquartz` (설치 후 맥북 재부팅 필수)
2. XQuartz 설정: 설정 -> 보안 탭에서 "Allow connections from network clients" 체크
3. 맥북 터미널에서 권한 허용: `xhost +localhost`
4. 도커 실행 시 환경 변수 추가: 도커를 실행할 때 `-e DISPLAY=host.docker.internal:0` 옵션을 넣어줘야 합니다.

### 도커 실행 시 설정 (컨테이너를 새로 띄울 때)
```
docker run -it \
  -e DISPLAY=host.docker.internal:0 \
  -v /tmp/.X11-unix:/tmp/.X11-unix \
  --name my_ros_container \
  ... (나머지 설정)
```
 -e DISPLAY=host.docker.internal:0: 도커에게 "화면은 호스트 머신의 0번 디스플레이를 사용해"라고 알려줍니다. \
 -v /tmp/.X11-unix:/tmp/.X11-unix: 리눅스 GUI 통신 소켓을 공유합니다.

### 현재 컨테이너 내부에서 테스트하는 법
```
# 테스트 툴 설치
apt-get update
apt-get install x11-apps -y

# 화면 띄우기 테스트
export DISPLAY=host.docker.internal:0
xeyes
```

### host.docker.internal:0 가 무엇인가?
도커 컨테이너는 원래 외부 세계와 단절된 '나만의 방' 같은 곳입니다. 그래서 컨테이너 안에서 localhost를 치면 **"내 방(컨테이너)의 안쪽"** 을 의미하게 됩니다.

하지만 우리는 **"내 방이 아니라 내 맥북(호스트)에 있는 XQuartz 화면을 보고 싶어!"** 라고 해야 하죠. 이때 쓰는 주소가 바로 **host.docker.internal** 입니다.

쉽게 말하면: 도커가 자동으로 **"이 주소는 내 집(맥북) 주소야!"** 라고 알려주는 일종의 DNS 별명입니다. 그래서 맥북에서 실행 중인 XQuartz를 찾아갈 수 있게 해줍니다. 

### xhost +localhost를 실행했는데도 turtlesim 창이 안 뜬다면?
컨테이너 생성 시 **볼륨 공유(-v)**를 하지 않았기 때문일 가능성이 큽니다.

상황: 컨테이너 내부에서 "화면 정보를 host.docker.internal:0으로 보내라"고 지시해도, 정작 맥북과 데이터를 주고받을 **'통로'(/tmp/.X11-unix)** 가 연결되어 있지 않으면 데이터가 허공으로 사라집니다.
